1 프롬프트 설계 원칙

1.1 4대 원칙

  1. 범용성: 다양한 상황에서도 동작, 불특정 다수의 사용자 대상 다양한 상황과 과제에 적용 가능한 프롬프트의 결과를 내야 한다.
  2. 목적 지향성: 특정 사용자 목적에 세분화된 기능을 제공해야 한다.
  3. 일관성: 여러가지 LLM 모델을 사용하더라도, 생성 할 때마다 결과가 바뀌면 안된다. 어느정도 스펙트럼 안에서 일정한 결과 유지.
  4. 경제성: 단어, 토큰수 운영 비용 API 비용 절감

1.2 Scaffolding Prompting

  • 비계식 설계
  • 인간의 학습 이론(ZPD)을 적용
  • 외국어를 학습하는 사람들의 학습 능력을 최대한 끌어올리려면, 최대한 쉬운 단계부터 어려운 단계까지 차츰차츰 비계식으로 올라가야 한다.
  • 단어 -> 문장 -> 문단 완성을 하는 단계별 학습 방식 사용
  • 프롬프트 설계를 할떄도 이 방식을 사용하면 정말 잘 설계를 할 수 있다.
  • Understand → Ideation → Analysis → Apply → Evaluate → Generalize
  • 풀어야될 문제(요구사항)에 대한 이해
  • 문제에 대한 이해를 바탕으로 여러가지 프롬프트를 만드는 아이디에이션 진행.
  • 아이디에이션을 통해 구현된 프롬프트를 분석.
  • 실제로 도입, 적용, 평가
  • 일반화 할 수 있는 결과를 얻으면 배포 진행.
  • 단계적 사고구조 유도

1.3 핵심 5 Tips

1.3.1 Tip1. 최신 모델 사용

  • 최근까지 GPT 4/5
  • 최신모델을 사용하면 프롬프트가 조금 엉성해도 좋은 결과를 얻을 수 있다.
  • 그러나 GPT 3.5을 이용해 A 라는 기능을 4/5 만큼 끌어올릴 수 있다면, 이 방법이 좋은 방법이다.

1.3.2 Tip2. 명확한 동사로 지시

어떤 단어가 좀더 구체적, 또는 명확하게 표현이 되어있는지 확인 조금더 명확한 동작을 하는 단어가 있을 수 있다. 뭐를 나누다 라는 맥락이 없으면 뜻을 정확하게 이해하기가 어려운 언어이다. 예: 분석해, 정리해, 요약해, 작성해 등 –> 정확하게 무엇을 하려고 하는지 맥락 없어도 이해 가능하다.

1.3.3 Tip3. 프롬프트를 구조화하자

구조화라 함은 어떻게 요소별로 나눈다던지 섹션별로 컴포지션을 나누어서, 언어모델이 읽기 편한 구조를 만든다. 대표적으로 2가지 방법(Markdown, 코드 프롬프트)이 있다. 프롬프트를 구조화하면 LLM이 입력을 더 명확하게 해석할 수 있고,
작업 목적·규칙·입력 데이터·출력 형식을 혼동 없이 처리할 수 있다.
구조화를 위한 핵심 기법은 Markdown, 코드 프롬프트, XML 태그이다.

  1. Markdown 기호 사용하기 (Headers, Lists, Emphasis)

Markdown은 LLM이 가장 잘 해석하는 구조적 문법 중 하나이다.
섹션을 명확히 나누고, 각 부분을 시각적으로 구분해줌으로써
모델의 오류를 줄이고 반복 작업 시 유지보수성이 높아진다.

1.1 헤더(Headers) 활용 헤더는 프롬프트의 구성을 크게 나누는 데 사용한다. 샾 개수만큼 주 제목, 부제목, 소제목 등의 계층적 우선순위 인덱스를 부여할 수 있다.

# 역할(Role)
너는 스포츠 기사 분석 엔진이다.

## 작업(Task)
아래 텍스트에서 장소(도시, 국가, 축구 클럽)를 모두 추출해라.

## 출력 형식(Format)
- "장소: A, B, C" 형태의 한 줄 출력
- 설명 금지

## 텍스트(Text)
{{TEXT}}

1.2 목록(Lists) 활용 하이픈이나, 번호를 이용 –> 수행할 순서가 있는 프롬프트에 사용한다. 절차, 규칙, 조건을 번호 목록으로 정리하면 LLM이 하나씩 적용하기 쉽다.

다음 규칙을 따라 작업을 수행해라.
1. 사람 이름은 제외한다.
2. 축구 클럽은 장소로 포함한다.
3. 결과는 쉼표로 구분한다.
4. 설명 문장은 쓰지 않는다.

1.3 강조(Emphasis) 사용

중요 조건을 굵게 또는 기울임으로 표시해 모델의 우선순위를 높인다. 실제로 이탤릭, 볼드체로 표현은 되지 않는다. 언어모델에게 일반 줄글의 텍스트가 아닌 다른 요소임을 강조.

- 반드시 **한국어**로 작성하라.
- 출력은 **한 줄**로만 작성한다.
- *설명 문장은 포함하지 않는다.*
  1. 코드 프롬프트(Code Prompting)

구조화를 좀더 구체적으로 하는 방식에 대해서 소개. AdaPlanner 논순 발췌, 구글의 리서처 분들이 많이 사용한다. 파이썬 언어를 사용해서 언어 모델이 정확히 수행 해야 하는 내용을 작성. 자연어 대신 코드 형식으로 의도를 명세하면 애매함이 줄어들고 LLM이 구조적으로 이해해 할루시네이션이 감소한다.

# 역할 정의
role = "정보 추출 엔진"

# 작업 정의
task = "텍스트에서 '장소명'을 추출해 정렬된 리스트로 반환"

# 출력 형식 정의
output = {
    "places": ["string"]
}

코드 방식의 장점:

  • 논리 구조가 명확하게 표현됨
  • 조건과 출력 형식이 혼동되지 않음
  • 정형화된 포맷을 자주 재사용 가능
  1. XML 태그 기반 구조화 (Claude 최적화)

엔트로픽 프롬팅 쿡북 참조하면 나오는 내용. 문장 내 구역 나눔, 해당 구적을 강조하는 효과, 반복작업에서 프롬프트 수정과 편집/용이, 언어모델의 처리 속도 향상 XML 태그는 텍스트 내 특정 영역을 명확히 구분해 Claude뿐 아니라 대부분의 LLM에서 구조적 파싱을 돕는 강력한 방식이다.

이 방식의 장점은 탬플릿이 가능하다.

  • 태그로 영역 지정
<email>
Show up at 6AM because I say so.
</email>

Make this email more polite.

LLM은 <email>...</email> 내부 텍스트만 변환하면 된다는 것을 명확히 인식한다.

  • 데이터 템플릿 예시
<animal>{{ANIMAL}}</animal>

입력만 바꿔 여러 데이터를 동일한 템플릿에 적용할 수 있다.

예:

<animal>Cow</animal>
<animal>Dog</animal>
<animal>Seal</animal>

장점:

  • 구역별 의미가 명확
  • 반복 작업에서 수정이 쉽고 안정적
  • 모델이 태그 단위로 파싱하기 때문에 오류 감소
  1. 구조화 기법이 필요한 이유
  • 명령어, 입력 데이터, 출력 형식을 명확히 구분해 모델의 해석 오류를 줄인다.
  • 규칙을 시각적으로 구분하여 작업 수행 품질의 일관성과 재현성을 높인다.
  • 재사용 가능한 템플릿이 되어 프롬프트 제작 비용과 시간 절감에 기여한다.

1.3.4 Tip4. 단문 중심의 간결한 문장을 쓰자.

  • 한국어는 짧고 명확하게 작성
  • 톤·형식·분량을 구체적으로 지시
- 나쁜 예
5월의 봄을 주제로 시를 써주고 짧게
작성해줘 봄의 느낌을 살려봐.
결과: 오월의 푸른 하늘 아래
새싹이 희망을 노래하네.
아침 이슬에 빛나는 꽃들,
봄바람 따라 춤추는 나비.
모든 거리는 축제가 되어.

- 좋은 예
너는 시인이야. 한국의 계절, 봄을 상상해봐.
5월이야. 다양한 꽃이 여기저기 폈어.
그 분위기에 어울리는 짧은 시를 5줄 이내로
작성해줘. 한글 고유의 멋이 시에 드러나면 좋겠어.
결과: 봄바람, 너른 들녘을 스치며
꽃잎들 춤추듯 흩날리네
산들산들 부는 바람에
눈부신 햇살 아래
은은한 꽃 향기에 취하노라

1.3.5 Tip5. 하지 말 것(X) → 할 것(O)

  • 부정 지시보다 긍정적 행동 지시가 안정적 구체적인 내용을 단문 형식으로 제공함과 언어모델에게 추측을 해야 하는 여지를 주면 안된다.
- 나쁜 예
다음은 에이전트와 고객의 대화이다.
고객에게 아이디와 비밀번호를 묻지
않는다. 반복하지 않는다.

- 좋은 예
다음은 에이전트와 고객과의 대화내용이다.
챗봇은 고객의 대화에서 드러난 문제를 진단하고 해결책을
제시해야 한다. 사용자 아이디와 비밀번호같은 개인정보를
묻지 않는 대신,
사용자가 www.samplewebsite.com/help/faq에서 관련
사항을 찾도록 안내해라.

1.4 프롬프트 제작에 100% 절대적인 것은 없다.

  • 절대 나와서는 안되는 조건을 적용하기
  • 되도록 LLM 이 추론을 할 수 없게 구체적으로 주기
  • 긍정문을 쓰되 절대 나와서 안되는 문구들은 2중 부정을 사용

2 LLM의 한계와 프롬프트 엔지니어링 필요성

2.1 LLM 한계

  • 할루시네이션
  • 편향성
  • 근거 없는 출처 생성
  • 수학 및 추론 한계
  • Prompt Injection(프롬프트 해킹)

2.2 프롬프트 엔지니어링이 필요한 이유

  • 정확성 향상
  • 오류 및 편향 감소
  • 모델 행동 제어
  • 사용자 경험 개선
  • 비용 절감
