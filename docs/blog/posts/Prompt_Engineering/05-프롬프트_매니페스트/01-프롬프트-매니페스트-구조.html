1 프롬프트를 왜 “매니페스트”로 관리하는가?

1.1 프롬프트 하나짜리 챗봇의 한계

처음 LLM 챗봇을 만들면 보통 이런 코드에서 시작한다.

prompt = """
당신은 고객 상담 전문가입니다.
다음 질문에 답하세요.
질문: {question}
답변:
"""

이것으로 PoC(개념 증명)는 통과한다. 하지만 운영 환경에 들어가면 문제가 생긴다.

상황 문제
답변 품질 개선을 위해 규칙 추가 프롬프트가 수백 줄로 비대해짐
한국어/영어 동시 지원 프롬프트 파일 복사 후 수동 번역 관리
few-shot 예시 추가/제거 거대한 문자열 안에서 예시를 찾아 수정
A/B 테스트 어떤 버전이 어떤 규칙을 포함하는지 추적 불가
여러 에이전트가 공통 규칙 공유 동일 내용을 여러 파일에 복붙

매니페스트(Manifest)는 이 문제를 해결하기 위한 “프롬프트의 목차”이다. 하나의 거대한 프롬프트 대신, 역할별로 분리된 YAML 파일들을 매니페스트가 순서대로 조합한다.

1.2 핵심 비유: 책의 목차

매니페스트 = 목차 (Table of Contents)
  ├── entry = 서론 (역할, 상황, 경계)
  ├── skill 1 = 1장 (질문 라우팅 규칙)
  ├── skill 2 = 2장 (답변 구조 규칙)
  ├── skill 3 = 3장 (인용 규칙)
  └── skill 4 = 부록 (few-shot 예시)

LLM에게 전달되는 최종 프롬프트는 이 “장(chapter)”들을 순서대로 이어붙인 결과물이다.

2 예시 프로젝트의 디렉토리 구조

이 글에서는 e-커머스 고객 상담 챗봇을 예시 도메인으로 사용한다. prompts/ 디렉토리를 열어보자.

prompts/
├── chatbot/                        # 고객 상담 챗봇 에이전트
│   ├── _manifest.yml               # 루트 매니페스트 (언어 분기)
│   ├── en/                          # 영어
│   │   ├── _manifest.yml            # 영어 매니페스트
│   │   ├── chatbot-cs-rules.yml                      # entry (핵심 역할)
│   │   ├── chatbot-response-structure.yml            # skill
│   │   ├── chatbot-answer-guide.yml                  # skill
│   │   ├── chatbot-scope-and-format.yml              # skill
│   │   ├── chatbot-citation-rules.yml                # skill
│   │   └── chatbot-cs-few-shot.yml                   # few-shot
│   ├── ko/                          # 한국어 (en과 동일 구조)
│   │   ├── _manifest.yml
│   │   └── ... (동일 파일명, 한국어 내용)
│   └── shared/                      # 언어 무관 공통 규칙
│       ├── chatbot-question-routing.yml
│       └── chatbot-product-catalog-rules.yml
│
├── order_tracker/                   # 주문 추적 에이전트
│   ├── _manifest.yml
│   ├── ko/
│   │   ├── _manifest.yml
│   │   ├── order-tracker-rules.yml                   # entry
│   │   ├── output-format.yml                         # skill
│   │   ├── few-shot-single-order.yml                 # few-shot
│   │   ├── few-shot-bulk-orders.yml                  # few-shot
│   │   ├── few-shot-refund.yml                       # few-shot
│   │   └── few-shot-exchange.yml                     # few-shot
│   └── shared/
│       ├── order-status-schema.yml
│       ├── shipping-rules.yml
│       └── input-routing.yml
│
└── faq_responder/                   # 단일 파일 (매니페스트 미사용)
    └── faq_responder.yml

주목할 점:

  • _manifest.yml: 파일명 앞의 _는 “이 파일은 직접 프롬프트가 아니라 목차(매니페스트)이다”라는 관례
  • shared/: 한국어/영어 모두 공유하는 규칙은 여기에 한 번만 작성
  • faq_responder/: 단순한 에이전트는 매니페스트 없이 단일 파일로도 동작

3 매니페스트 파일 구조 상세

3.1 2단계 매니페스트: 루트 -> 언어별

이 프로젝트의 매니페스트는 2단계 계층으로 설계되어 있다.

[루트 매니페스트] ──> [한국어 매니페스트] ──> entry + skills
                 └──> [영어 매니페스트]   ──> entry + skills

3.1.1 1단계: 루트 매니페스트 (언어 분기기)

prompts/chatbot/_manifest.yml
_type: "manifest"
domain: "Customer Service"
name: "chatbot-customer-service"
description: "e-커머스 고객 상담 챗봇의 루트 매니페스트"
version: "1.0.0"
updated_at: "2026-03-16"
tags: ["customer-service", "ecommerce", "qa-chatbot"]

locales:                        # <-- 언어별 매니페스트 참조
  ko: ko/_manifest.yml
  en: en/_manifest.yml

default_locale: "ko"            # <-- 언어 미지정 시 기본값

필드 설명:

필드 역할
_type 파일 종류 식별. "manifest"이면 목차 파일
domain 비즈니스 도메인 (검색/필터링용)
name 매니페스트 고유 식별자
version 시맨틱 버전 (1.0.0)
locales 언어 코드 -> 하위 매니페스트 경로 매핑
default_locale 언어 미지정 시 사용할 기본 언어

핵심: 루트 매니페스트는 프롬프트 내용을 전혀 포함하지 않는다. 오직 “어떤 언어 매니페스트를 불러올 것인가”만 결정한다.

3.1.2 2단계: 언어별 매니페스트 (entry + skills 조합)

prompts/chatbot/ko/_manifest.yml
_type: "manifest"
domain: "Customer Service"
name: "chatbot-customer-service"
lang: "ko"
description: "e-커머스 고객 상담 챗봇의 한국어 프롬프트 매니페스트"
version: "1.0.0"
updated_at: "2026-03-16"
tags: ["customer-service", "ecommerce", "qa-chatbot", "ko"]

entry: chatbot-cs-rules.yml                      # <-- 핵심 프롬프트 (1개)

skills:                                          # <-- 부가 규칙 (순서대로 결합)
  # 공유 스킬 (언어 무관)
  - name: ../shared/chatbot-question-routing.yml
    description: "질문 유형별 라우팅 결정 트리를 정의한다."
  - name: ../shared/chatbot-product-catalog-rules.yml
    description: "상품 카탈로그 스키마, 카테고리 구분, 검색 규칙을 정의한다."

  # 한국어 전용 스킬
  - name: chatbot-response-structure.yml
    description: "답변의 서론/본문/결론/관련질문 구조와 메타 표현 금지 규칙을 정의한다."
  - name: chatbot-answer-guide.yml
    description: "한국어 답변 원칙, 환불 요청 감지, 도메인 지식을 정의한다."
  - name: chatbot-scope-and-format.yml
    description: "범위 외 질문 처리 규칙과 출력 서식 규칙을 정의한다."
  - name: chatbot-citation-rules.yml
    description: "검색된 문서(Context) 활용법과 인라인 citation 규칙을 정의한다."
  - name: chatbot-cs-few-shot.yml
    description: "고객 상담 챗봇의 few-shot 예시 4개를 정의한다."

핵심 구조:

  • entry: 프롬프트의 “본체”. 역할(role), 상황(context), 경계(boundaries)를 정의한다. 반드시 1개.
  • skills: 프롬프트에 추가되는 규칙 모듈. 0개 이상, 순서 중요.
  • ../shared/ 경로로 언어 공통 스킬을 참조할 수 있다.

3.2 프롬프트 파일의 3가지 유형

3.2.1 유형 1: entry (핵심 프롬프트)

prompts/chatbot/en/chatbot-cs-rules.yml
_type: "prompt"
name: "chatbot-cs-rules"
description: "Core prompt defining the role, context, and behavioral boundaries"

template: |
  <role>
  You are a customer service specialist
  for an e-commerce platform selling electronics.
  </role>

  <context-situation>
  - Customers may ask about orders, returns, and product specs.
  - You have access to the product catalog and order history.
  </context-situation>

  <boundaries>
  CAN:
  - Explain return/refund policies
  - Look up order status and shipping info
  - Recommend alternative products

  CANNOT:
  - Process payments or issue refunds directly
  - Make up information not in the provided context
  - Share other customers' personal data
  </boundaries>

  #Chat History (recent conversation context):
  {chat_history}

  #Context:
  {context}

  #Question:
  {question}

  #Answer:

input_variables: ["question", "context", "chat_history"]

포인트:

  • _type: "prompt" – entry 전용 유형
  • template 안의 {question}, {context}, {chat_history}는 런타임에 실제 값으로 치환되는 변수
  • input_variables는 이 프롬프트가 필요로 하는 변수 목록 (LangChain PromptTemplate에 전달)
  • XML 태그(<role>, <boundaries>)로 구간을 명확히 구분

3.2.2 유형 2: skill (규칙 모듈)

prompts/chatbot/en/chatbot-response-structure.yml
_type: "skill"
name: "chatbot-response-structure"
description: "Defines the greeting/body/closing/follow-up response structure."
version: "1.0.0"
updated_at: "2026-03-16"

template: |
  <response-structure>
  Answers follow the logical structure of
  greeting / body / closing / follow-up suggestions.

  <fix-meta-expression>
  WRONG: "Greeting: Hello, thank you for contacting us..."
  CORRECT: Start answering directly with natural sentences
  </fix-meta-expression>

  <response-greeting>
  - Use a brief, friendly greeting only on the first interaction.
  - For follow-up messages, skip the greeting and go straight
    to the answer.
  </response-greeting>
  </response-structure>

포인트:

  • _type: "skill" – entry의 뒤에 이어붙여지는 규칙 조각
  • input_variables가 없다 – skill은 변수를 새로 도입하지 않고, entry의 변수를 그대로 사용
  • 하나의 skill = 하나의 관심사 (응답 구조, 인용 규칙, 범위 제한 등)

3.2.3 유형 3: few-shot-examples (예시 모음)

prompts/chatbot/en/chatbot-cs-few-shot.yml
_type: "few-shot-examples"
domain: "Customer Service"
name: "chatbot-cs-few-shot"
description: "Few-shot examples for the customer service chatbot."
version: "1.0.0"
updated_at: "2026-03-16"
tags: ["customer-service", "ecommerce", "few-shot", "en"]

examples:
  - id: "example-01"
    title: "Order Status Inquiry"
    question: "My order #A12345 has not arrived yet. It's been 7 days."
    answer: |
      I can see your order #A12345 was shipped on March 10th
      via express delivery. The current tracking status shows
      it is at the local distribution center.

      Expected delivery: March 18th.
      If it doesn't arrive by then, please contact us again
      and we'll initiate a delivery investigation.

  - id: "example-02"
    title: "Return Policy Question"
    question: "Can I return a laptop I bought 20 days ago?"
    answer: |
      Our standard return window is 14 days from delivery.
      Since 20 days have passed, a regular return isn't possible.

      However, if the product is defective, you may be eligible
      for a warranty claim. Would you like me to check
      the warranty options for your laptop?

포인트:

  • _type: "few-shot-examples" – 구조화된 Q&A 예시
  • examples 배열 안에 id, title, question, answer 형태로 정리
  • 로더가 이 배열을 자동으로 ## Few-shot 예시 텍스트 블록으로 변환

4 코드: 매니페스트를 어떻게 로딩하는가

4.1 로더의 전체 흐름

4.2 매니페스트 로더 구현 예시

아래는 위 플로우차트를 구현한 예시 코드이다. 실제 프로젝트에서도 이와 동일한 패턴을 사용한다.

prompts/loader.py -- 메인 진입점
from pathlib import Path
import yaml
from langchain.prompts import PromptTemplate


def _load_yaml(path: Path) -> dict:
    """YAML 파일을 읽어 dict로 반환한다."""
    with open(path, encoding="utf-8") as f:
        return yaml.safe_load(f)


def has_manifest(directory: Path) -> bool:
    """디렉토리에 _manifest.yml이 있는지 확인한다."""
    return (Path(directory) / "_manifest.yml").exists()


def load_manifest_prompt(
    manifest_path: Path,
    locale: str | None = None,
) -> PromptTemplate:
    """매니페스트 파일을 읽고, entry + skills 를 결합한
    PromptTemplate을 반환한다."""

    manifest_path = Path(manifest_path)
    base_dir = manifest_path.parent
    manifest = _load_yaml(manifest_path)

    # 1단계: 루트 매니페스트에 locales가 있으면 언어 분기
    if "locales" in manifest:
        locale = locale or manifest.get("default_locale", "ko")
        locale_ref = manifest["locales"].get(locale)
        if locale_ref is None:
            raise ValueError(
                f"지원하지 않는 locale: '{locale}'"
            )
        locale_manifest_path = base_dir / locale_ref
        return _load_locale_manifest(locale_manifest_path)

    # locales가 없으면 이 매니페스트 자체가 언어별 매니페스트
    return _load_locale_manifest(manifest_path)
prompts/loader.py -- entry + skills 결합
def _load_locale_manifest(
    manifest_path: Path,
) -> PromptTemplate:
    """언어별 매니페스트를 로드하여 PromptTemplate을 조합한다."""

    base_dir = manifest_path.parent
    manifest = _load_yaml(manifest_path)

    # -- entry 로드 (핵심 프롬프트) --
    entry_path = base_dir / manifest["entry"]
    entry = _load_yaml(entry_path)
    core_template = entry["template"]
    input_variables = entry.get("input_variables", [])

    # -- skills 순서대로 로드 & 결합 --
    for skill_ref in manifest.get("skills", []):
        skill_path = (base_dir / skill_ref["name"]).resolve()
        skill = _load_yaml(skill_path)

        if "template" in skill:
            # skill 유형: 텍스트를 그대로 이어붙임
            core_template += "\n" + skill["template"]
        elif "examples" in skill:
            # few-shot 유형: Q&A 포맷으로 렌더링
            core_template += "\n" + _render_few_shot(
                skill["examples"]
            )

    return PromptTemplate(
        template=core_template,
        input_variables=input_variables,
    )
prompts/loader.py -- few-shot 렌더링
def _render_few_shot(examples: list[dict]) -> str:
    """few-shot examples 리스트를 프롬프트 텍스트로 변환한다."""

    lines = ["\n  ## Few-shot 예시\n"]
    for ex in examples:
        lines.append(
            f"  ### 예시: {ex.get('title', ex.get('id', ''))}"
        )
        lines.append(f"  **Q:** {ex['question']}")
        lines.append(f"  **A:**\n{ex['answer']}")
    return "\n".join(lines)

4.3 페이지에서 호출하는 방법

각 Streamlit 페이지는 다음 패턴으로 프롬프트를 로딩한다.

pages/order_tracker.py -- 단순 호출 예시
PROMPTS_DIR = (
    Path(__file__).parent.parent / "prompts" / "order_tracker"
)

def _load_prompt():
    """매니페스트가 있으면 매니페스트 로딩, 없으면 단일 파일 로딩"""
    if has_manifest(PROMPTS_DIR):
        return load_manifest_prompt(
            PROMPTS_DIR / "_manifest.yml"
        )
    return load_prompt(
        str(PROMPTS_DIR / "order_tracker.yml"),
        encoding="utf-8",
    )
pages/chatbot.py -- RAG 체인 생성 예시
def create_rag_chain(retriever, model_name, prompt_path):
    prompt_path = Path(prompt_path)

    # 경로가 디렉토리이면 -> _manifest.yml 자동 탐색
    if prompt_path.is_dir():
        prompt = load_manifest_prompt(
            prompt_path / "_manifest.yml"
        )
    # 경로가 _manifest.yml 자체이면 -> 바로 로드
    elif prompt_path.name == "_manifest.yml":
        prompt = load_manifest_prompt(prompt_path)
    # 같은 디렉토리에 매니페스트가 있으면 -> 매니페스트 우선
    elif has_manifest(prompt_path.parent):
        prompt = load_manifest_prompt(
            prompt_path.parent / "_manifest.yml"
        )
    # 매니페스트 없음 -> 단일 파일 로드
    else:
        prompt = load_prompt(
            str(prompt_path), encoding="utf-8"
        )

설계 의도: 매니페스트를 도입하기 전의 단일 파일 프롬프트와 하위 호환된다. _manifest.yml이 없으면 기존 방식으로 동작한다.

5 매니페스트의 실전 활용 패턴

5.1 패턴 1: 다국어 지원

루트 _manifest.yml
  └── locales:
        ko: ko/_manifest.yml  -->  한국어 entry + 한국어 skills
        en: en/_manifest.yml  -->  영어 entry + 영어 skills
  • 언어 추가 = ja/ 폴더 생성 + 루트 매니페스트에 ja: ja/_manifest.yml 한 줄 추가
  • 코드 변경 없이 locale="ja" 파라미터만 전달하면 됨

5.2 패턴 2: 공유 스킬 (shared)

# ko/_manifest.yml 에서
skills:
  - name: ../shared/chatbot-question-routing.yml    # 언어 무관
  - name: chatbot-answer-guide.yml                   # 한국어 전용
  • 질문 라우팅 로직은 언어와 무관 -> shared/에 한 번만 작성
  • 답변 작성 규칙은 언어별로 다름 -> 각 언어 폴더에 별도 작성
  • 중복 제거와 일관성 유지를 동시에 달성

5.3 패턴 3: 버전 관리 및 A/B 테스트

설정 파일에서 프롬프트 버전을 지정할 수 있다.

configs/default.yaml
prompt:
  template_name: chatbot_cs_rules
  version: v1                    # <-- v2로 바꾸면 다른 프롬프트 로드
  system_message_override: null

버전 해석 로직:

version == "v1"  -->  prompts/{agent}/{template_name}.yml
version == "v2"  -->  prompts/{agent}/{template_name}_v2.yml
  • 운영 중 프롬프트 개선 시: v2 파일을 만들고, 설정만 바꿔서 테스트
  • 문제 발생 시: 설정을 v1으로 되돌리면 즉시 롤백

5.4 패턴 4: 매니페스트 없는 단순 에이전트와의 공존

# 매니페스트가 있으면 매니페스트 로딩
if has_manifest(PROMPTS_DIR):
    return load_manifest_prompt(PROMPTS_DIR / "_manifest.yml")
# 없으면 단일 YAML 파일 로딩
return load_prompt(str(PROMPTS_DIR / "agent.yml"))
  • faq_responder/처럼 규칙이 단순한 에이전트는 매니페스트 없이 동작
  • 나중에 규칙이 복잡해지면 _manifest.yml을 추가하기만 하면 자동 전환

6 최종 조합 결과: LLM에게 전달되는 프롬프트

매니페스트 로딩이 완료되면 LLM에게는 다음 순서로 결합된 하나의 긴 프롬프트가 전달된다.

+---------------------------------------------+
|  entry: chatbot-cs-rules                     |  <role>, <boundaries>,
|  (역할, 상황, 경계, 변수 슬롯)                  |  {question}, {context}
+---------------------------------------------+
|  skill 1: question-routing (shared)          |  질문 분류 결정 트리
+---------------------------------------------+
|  skill 2: product-catalog-rules (shared)     |  상품 카탈로그 규칙
+---------------------------------------------+
|  skill 3: response-structure                 |  답변 구조 (인사/본문/마무리)
+---------------------------------------------+
|  skill 4: answer-guide                       |  답변 원칙
+---------------------------------------------+
|  skill 5: scope-and-format                   |  범위 외 처리
+---------------------------------------------+
|  skill 6: citation-rules                     |  인용 규칙
+---------------------------------------------+
|  skill 7: few-shot examples                  |  예시 Q&A 4개
+---------------------------------------------+

이 결합된 텍스트가 LangChain PromptTemplate으로 감싸져서, 실행 시 {question}, {context}, {chat_history}에 실제 값이 채워진다.

7 매니페스트 도입 체크리스트

직접 프로젝트에 매니페스트 구조를 도입할 때 참고할 체크리스트:

단계 작업 산출물
1 기존 프롬프트에서 역할/경계 분리 entry.yml
2 반복되는 규칙을 skill 파일로 추출 skill-*.yml
3 Q&A 예시를 few-shot 파일로 분리 few-shot-*.yml
4 언어별 폴더 생성 (ko/, en/) 디렉토리 구조
5 언어 무관 규칙을 shared/로 이동 shared/*.yml
6 언어별 _manifest.yml 작성 entry + skills 목록
7 루트 _manifest.yml 작성 locales 매핑
8 로더 코드에 매니페스트 분기 추가 has_manifest() 체크
9 기존 단일 파일과 하위 호환 확인 폴백 테스트

8 메타데이터 필드 레퍼런스

모든 YAML 파일에서 사용되는 메타데이터 필드 정리:

필드 필수 대상 설명
_type Y 전체 manifest, prompt, skill, few-shot-examples
name Y 전체 고유 식별자
description Y 전체 파일의 목적 설명
domain N 전체 비즈니스 도메인
version N 전체 시맨틱 버전 (1.0.0)
updated_at N 전체 최종 수정일 (ISO 날짜)
author N 전체 작성자
tags N 전체 검색용 키워드 배열
lang N 언어별 언어 코드 (ko, en)
locales N 루트 매니페스트 언어 -> 하위 매니페스트 경로
default_locale N 루트 매니페스트 기본 언어
entry Y 언어별 매니페스트 핵심 프롬프트 파일명
skills N 언어별 매니페스트 스킬 목록 (name + description)
template Y prompt, skill 프롬프트 텍스트
input_variables Y prompt (entry) 런타임 치환 변수 목록
examples Y few-shot-examples Q&A 예시 배열

9 정리

항목 단일 파일 방식 매니페스트 방식
구조 하나의 긴 YAML/텍스트 entry + skills 분리
다국어 파일 복사 + 수동 관리 locales 자동 분기
공통 규칙 복붙 shared/에서 참조
수정 범위 전체 파일 건드림 해당 skill만 수정
A/B 테스트 파일 전체 복제 설정에서 version만 변경
하위 호환 - _manifest.yml 없으면 기존 방식
적합한 상황 규칙이 단순한 에이전트 규칙이 복잡하고 다국어 지원 필요

매니페스트 구조의 핵심은 “프롬프트를 코드처럼 관리한다”는 것이다. 함수를 분리하듯 프롬프트를 역할별로 분리하고, import하듯 매니페스트에서 조합한다. 이렇게 하면 프롬프트 수정이 코드 수정만큼 안전하고 추적 가능해진다.
