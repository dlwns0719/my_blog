1 배경

ML 분류 모델의 학습 데이터를 생성하는 프로젝트를 진행했다. RAG 기반 챗봇의 시스템 프롬프트를 설계하여, 14개 카테고리에 대해 각 200개 이상(총 2,800~3,000개)의 데이터를 자동 생성하는 것이 목표였다.

1주일간 프롬프트를 v0.1부터 v0.6까지 반복 수정했지만 만족할 만한 품질을 달성하지 못했고, 결국 도메인에 대한 깊은 이해를 가진 동료가 30분 만에 v0.7 프롬프트를 작성하여 프로젝트를 완성했다.

이 글은 그 과정에서 배운 프롬프트 엔지니어링 교훈을 정리한 것이다.

2 프로젝트 구조

2.1 하이브리드 엔진 아키텍처

입력(논리명 또는 설명)을 3개의 엔진이 동시에 처리하는 구조:

  • 약어 생성기: 물리명, 설명 출력
  • ML 분류 모델: 도메인 그룹 출력 (이번 프로젝트의 대상)
  • LLM: 논리명 또는 물리명 설명 출력

ML 분류 모델을 학습시키기 위한 데이터가 필요했고, RAG 챗봇을 이용해 데이터를 증강하는 Agent의 시스템 프롬프트를 설계하는 것이 나의 과제였다.

2.2 데이터 요구사항

  • 14개 카테고리, 각 카테고리당 최소 200개
  • Good 데이터(정상 케이스)와 Edge 데이터(경계 케이스) 구분
  • 특히 edge case 데이터의 라벨링이 어려운 영역에 대한 데이터가 중요

3 프롬프트 버전별 변화와 문제점

3.1 v0.1~v0.2 (약 180줄) - “다 봐라” 수준

3.1.1 작성한 프롬프트

- 데이터를 생성하기 전에 9장(표준 용어사전)을 검색해서 확인하고 이를 정확히 지켜서 생성한다.
- 데이터를 생성하기 전에 8장(표준 도메인 사전)의 내용을 검색해서 확인하고 이를 정확히 지켜서 생성한다.
- 데이터를 생성하기 전에 7장(표준 단어사전 원칙)의 내용을 검색해서 확인하고 이를 정확히 지켜서 생성한다.

3.1.2 문제

  • “검색해서 확인하고 지켜라”는 사람에게도 막연한 지시인데, LLM에게는 더더욱 무의미하다
  • 각 장이 무엇을 담고 있고, 그중 어떤 부분이 이 작업에 핵심인지 구체화하지 않았다
  • 분류 단어와 수식어 단어의 구분에 대한 지시가 없었다
  • edge case를 만들기 위한 가이드라인이 부재했다

3.2 v0.3~v0.6 (약 475줄) - “문서 복붙” 수준

3.2.1 변화

  • 각 카테고리별 생성 가이드를 추가
  • 혼동 쌍(edge case) 정보를 일부 추가

3.2.2 여전한 문제

  • 원칙 문서의 내용을 그대로 복붙하여 시스템 프롬프트에 삽입
  • 구조화된 생성 절차(step-by-step)가 없어 LLM이 무엇부터 해야 할지 판단 불가
  • 허용 단어 목록이 명시적으로 제공되지 않아 LLM이 임의로 생성하는 문제 발생

3.3 v0.7 (약 465줄) - 도메인 전문가 작성

3.3.1 핵심 차이점

항목 v0.1~v0.6 v0.7
생성 절차 없음 7단계 순차 절차 명시
허용 단어 “검색해서 확인” 카테고리별 단어 목록 테이블로 제공
판별 기준 “원칙문서 참고” Q1~Q6 결정 트리 + 우선순위 규칙
수식어 가이드 없음 6개 카테고리 명시
분배 계획 없음 단어별 균등 분배 계획 후 생성
자체 검증 없음 6개 항목 체크리스트
반례 없음 나쁜 예 -> 좋은 예 비교 테이블

4 핵심 교훈

4.1 교훈 1: 시스템 프롬프트는 “엄격한 업무 지시서”이다

잘못된 이해: 시스템 프롬프트에 원칙 문서를 통째로 넣으면 LLM이 알아서 판단하겠지.

올바른 이해: RAG 시스템에서 시스템 프롬프트에 문서를 복붙하면 “이 텍스트만을 기준으로 동작하라”는 강제 명령이 된다. LLM이 검색(retrieval)을 통해 가져온 context를 자율적으로 해석할 여지를 없애버린다.

4.1.1 RAG에서 시스템 프롬프트의 올바른 역할

시스템 프롬프트 = 판단 기준 + 절차
RAG context    = 원칙 문서의 상세 내용 (검색 결과)
  • 시스템 프롬프트에는 무엇을 해야 하는지(절차)와 어떻게 판단해야 하는지(기준)를 넣는다
  • 원칙 문서의 원문은 RAG의 검색 대상(vector store)으로 남겨두고, 시스템 프롬프트에는 검색 결과를 어떤 관점으로 해석할지 지시한다
"8.5.8절의 우선순위 규칙을 적용하라" (O)
"8.5.8절 전체를 참고하라"           (X)

4.2 교훈 2: “구체적”이란 것의 실체

v0.7 프롬프트에서 배운 “구체적”의 4가지 차원:

4.2.1 절차의 구체성

(X) "데이터를 생성하라"
(O) "7단계를 순서대로 따르라. 다음 단계로 넘어가기 전에 현재 단계를 완전히 완료하라."

4.2.2 기준의 구체성

(X) "카테고리에 맞게 분류하라"
(O) "Q1~Q6 결정 트리를 적용하라. Q1: 보안/암호화가 필요한가? YES -> 보안 카테고리"

4.2.3 제약의 구체성

(X) "표준 단어를 사용하라"
(O) "목록에 없는 단어를 분류 단어로 임의 생성하지 않는다"

4.2.4 예시의 구체성

(X) "좋은 용어를 만들어라"
(O) "시약농도(X) -> 추출시약농도단위(O), 이유: 주제어/분류어 누락"

4.3 교훈 3: 도메인 이해 없이 프롬프트는 만들 수 없다

프로젝트에서 내가 인지하지 못한 것들:

  • 원칙 문서의 특정 절이 edge case 생성의 핵심 가이드라인이라는 사실
  • “형식 단어 여부” 항목이 분류 단어와 수식어 단어를 구분하는 핵심이라는 사실
  • 결과적으로 “어떤 부분을 봐야 하는지”조차 판단 불가

프롬프트 엔지니어링의 70%는 도메인 이해이다. LLM에게 지시를 내리려면 내가 먼저 그 업무를 이해해야 한다. “이 문서의 어떤 부분이 이 작업에 핵심인가?”를 판별하는 능력이 곧 프롬프트의 품질을 결정한다.

4.4 교훈 4: 결과를 검증할 수 없으면 프롬프트를 개선할 수 없다

생성된 결과를 보고 “이게 좋은 품질인지” 판단할 수 없었다. 이유는 분류 기준 자체를 이해하지 못했기 때문이다.

예를 들어 “시약농도”라는 용어가 생성되었을 때, 이것이 왜 잘못인지(주제어/분류어 누락) 인지하지 못하면 프롬프트를 어떻게 고쳐야 할지도 모른다.

결과가 나왔을 때 “이거면 됐지”가 아니라 “왜 이게 맞는지/틀린지”를 검증하는 루프가 프롬프트 개선의 핵심이다.

5 프롬프트 설계 패턴 5가지

v0.7에서 추출한, 실전에서 효과적인 프롬프트 설계 패턴들이다.

5.1 패턴 A: 절차 기반 구조 (Step-by-Step Procedure)

## 생성 절차
학습 데이터를 생성할 때는 아래 7단계 순서를 반드시 따른다.
다음 단계로 넘어가기 전에 현재 단계를 완전히 완료한다.

## 1단계: 클래스 정의 확인
## 2단계: 허용 단어 목록 확정
## 3단계: 수식어 맥락 목록 확정
## 4단계: 생성 분배 계획 수립
## 5단계: 정상 데이터 생성
## 6단계: 경계 데이터 생성
## 7단계: 자체 검증

효과: LLM이 “무엇부터 해야 하지?”라는 판단 없이, 각 단계를 순차적으로 수행하면 된다.

적용 상황: 복잡한 생성 태스크, 다단계 분석, 데이터 파이프라인 설계

5.2 패턴 B: 결정 트리 (Decision Tree)

| 질문 | YES | NO |
|------|-----|-----|
| Q1: 보안/암호화가 필요한가? | -> 보안 | Q2로 |
| Q2: 분류/범주화 목적인가?   | -> Q3으로 | Q4로 |
| Q3: 다중 분류가 필요한가?   | -> 코드 | -> 분류 |
| Q4: 사람이 읽는 명칭인가?   | -> 명칭 | Q5로 |
| Q5: 순수 숫자인가?         | -> 번호 | Q6으로 |
| Q6: 문자+숫자 조합인가?    | -> 식별 | -> 번호 |

효과: 모호한 판단을 if-else 분기로 환원. LLM이 경계 상황에서 명확한 경로를 따를 수 있다.

적용 상황: 분류 태스크, 모호한 카테고리 구분, 예외 처리 로직

5.3 패턴 C: 허용 목록 (Allowlist)

## 카테고리별 허용 단어 목록
목록에 없는 단어를 분류 단어로 임의 생성하지 않는다.

| 단어 | 형식 | 데이터타입 | 예시 |
|------|------|-----------|------|
| 일자 | YYYYMMDD | VARCHAR(8) | 20160420 |
| 시각 | HHMMSS | VARCHAR(6) | 143022 |

효과: LLM의 “창의성”이 오히려 해가 되는 영역에서 범위를 제한한다. 동시에 다른 영역(수식어 등)에서는 창의성을 허용한다.

적용 상황: 정해진 용어 체계가 있는 도메인, 코드 생성, 데이터 스키마 설계

5.4 패턴 D: 반례 기반 가이드 (Bad -> Good Examples)

| 나쁜 예 | 좋은 예 | 문제 |
|---------|---------|------|
| 시약농도 | 추출시약농도단위 | 주제어/분류어 누락 |
| 등록자 | 검체등록자성명 | 주제어/분류어 누락 |
| Ct값 | PCR전처리완료Ct값 | 약어 사용, 주제어 누락 |

효과: 추상적인 규칙을 구체적인 사례로 보여준다. LLM이 패턴을 학습하여 유사한 실수를 회피한다.

적용 상황: 명명 규칙, 코드 스타일, 문서 작성 가이드

5.5 패턴 E: 분배 계획 (Distribution Planning)

## 4단계: 생성 분배 계획 수립
- 전체 목표 개수(200개)를 허용 단어 수로 균등 분배한다.
- 각 단어별 목표 개수를 먼저 표 형태로 계획한 후 생성 단계로 넘어간다.
- 수식어 카테고리별로도 골고루 분배되도록 계획한다.

효과: “200개 만들어”라고만 하면 특정 패턴에 편중된다. 분배 계획을 먼저 세우게 함으로써 다양성을 확보한다.

적용 상황: 대량 데이터 생성, 테스트 케이스 설계, 학습 데이터 밸런싱

6 근본 원인 분석

6.1 왜 실패했는가

6.1.1 태스크 분해 실패

“데이터 증강 프롬프트를 만들어라”라는 과제를 받았을 때, 아래처럼 분해했어야 했다:

  1. 원칙 문서의 핵심 개념 파악
  2. 14개 카테고리 각각의 정의와 판별 기준 정리
  3. 카테고리 간 경계(edge case)가 모호한 영역 식별
  4. 용어 구성 규칙 정리
  5. 위 내용을 기반으로 생성 절차 설계
  6. 프롬프트 작성 -> 테스트 -> 결과 검증 -> 수정

실제 행동: 1~4를 건너뛰고 5~6만 반복. 결과의 품질을 판단할 기준이 없으니 “이거면 됐지”로 넘어감.

6.1.2 “왜?”를 묻지 않음

동료가 “문서의 특정 절을 보라”고 했을 때:

  • 나의 행동: 시킨 대로만 봄
  • 해야 했던 행동: “왜 이 부분이 중요하지?” -> 카테고리 간 경계가 모호한 케이스의 판별 기준이 여기 있기 때문 -> 결정 트리와 우선순위 규칙을 추출해야 함

지시를 받았을 때 “왜 이걸 보라고 했지?”, “왜 이 정보가 필요하지?”를 고민하지 않으면, 지시의 표면만 따라가게 된다.
