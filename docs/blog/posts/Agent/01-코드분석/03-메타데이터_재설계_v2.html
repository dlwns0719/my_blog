1 메타데이터 재설계: v1에서 v2로

1.1 v1 결과에 대한 반성

1편에서 12개 분산 에이전트로 2,499개 메타데이터를 추출했고, 2편에서 AST + LLM 하이브리드로 재현성을 99%까지 끌어올렸다.

그런데 추출된 메타데이터를 실제 Agent에 투입하려고 검토하면서 근본적인 문제를 발견했다.

1.1.1 1,100개 중 80%는 코드를 읽으면 바로 알 수 있는 정보였다

v1에서 추출한 원자적 메타데이터 예시:

# 파일 물리명 논리명 데이터 타입 값/기본값 의미
1 main.py –ui_json UI JSON Path Argument str (필수) 입력 JSON 파일 경로를 지정하는 커맨드라인 인자
2 main.py –output_folder Output Folder Path Argument str (필수) 산출물이 저장될 출력 폴더 경로
3 main.py –log_folder Log Folder Path Argument str (필수) 로그 파일이 저장될 폴더 경로

이 3개 항목은 main.py를 열면 즉시 확인 가능하다. Agent가 코드를 읽으면 자체적으로 파악할 수 있는 정보이므로, 별도 메타데이터로 관리할 이유가 없다.

이런 항목이 전체의 약 80%를 차지하고 있었다. Agent에게 실질적으로 도움이 되는 “코드만으로는 알 수 없는 맥락” 항목은 소수에 불과했다.

1.1.2 근본 질문

메타데이터란 무엇인가? Agent에게 정말 필요한 정보는 무엇인가?

이 질문에 제대로 답하지 않은 채 “많이 추출하면 좋겠지”라는 접근이 v1의 근본 문제였다.

1.2 메타데이터 정의 재정립

1.2.1 v1 정의 (모호)

“사람만 아는 지식”

이 정의는 추상적이어서, LLM이 코드에서 바로 읽을 수 있는 변수명이나 타입 힌트까지 메타데이터로 추출하게 만들었다.

1.2.2 v2 정의 (명확)

코드로부터 바로 알 수 없고, 코드를 올바르게 이해하고 수정하기 위해 반드시 필요한 맥락.

두 가지 조건을 동시에 만족해야 메타데이터이다:

조건 설명 불충족 시
코드로부터 바로 알 수 없다 Agent가 코드 전체를 읽어도 정확히 알 수 없는 정보 코드 자체로 충분 (메타데이터 불필요)
올바른 이해/수정을 위해 반드시 필요하다 이 정보 없이 코드를 수정하면 문제가 발생 있으면 좋지만 필수 아님 (메타데이터 불필요)

1.2.3 2단계 판별 기준

메타데이터 추출 시 매 항목마다 이 질문을 통과해야 한다:

[1단계] Agent가 코드를 전부 읽으면 이 정보를 정확히 알 수 있는가?
  → YES → 메타데이터 아님 (제외)
  → NO  → 2단계로

[2단계] 이 정보 없이 코드를 수정하면 문제가 생기는가?
  → YES → 메타데이터 (포함)
  → NO  → 메타데이터 아님 (제외)

이 필터를 적용하면 v1의 80%가 1단계에서 탈락한다. 변수 타입, CLI 인자명, 함수 시그니처 등은 코드를 읽으면 바로 알 수 있기 때문이다.

1.3 LLM이 코드를 보고도 모르는 것: 두 가지 유형

코드만으로 Agent가 정확한 답에 도달하지 못하는 이유는 두 가지다.

1.3.1 유형 1: 코드에 없는 정보 (단일 모듈 내)

코드를 아무리 읽어도 답할 수 없는 질문이 존재한다. 정보 자체가 코드 바깥(담당자 머릿속, 실험 결과, 규제 문서 등)에만 있다.

질문 LLM 응답 실제 정보 출처
“이 임계값 0.5는 왜 이 값인가?” “코드에서 확인할 수 없습니다” 담당자 경험/실험 데이터
“CI는 무슨 뜻인가?” “Continuous Integration으로 추정” 도메인 전문 용어 (Control Internal)
“왜 pickle 안 쓰고 SQLite인가?” “성능 때문으로 추정” 과거 pickle 버전 호환 장애 히스토리

해결: 모듈별 메타데이터 (도메인 지식, 설계 근거, 운영 경험)

1.3.2 유형 2: 코드에 있지만 추적이 끊기는 정보 (모듈 간)

정보는 코드에 존재하지만, 여러 모듈에 걸쳐 있어서 Agent가 추적 중간에 맥락을 놓치는 문제이다.

예를 들어, 파이프라인이 7개 모듈로 구성되어 있을 때:

모듈 A: 올리고 설계 조건 변경
  → 모듈 B: 설계 조건이 시퀀스 필터에 반영
    → 모듈 C: 필터된 시퀀스로 단일 OM 생성
      → 모듈 D: oligo_set을 읽어서 최적화
        → 모듈 E: 최적화 결과를 TSV로 출력
          → 모듈 F: TSV를 읽어서 다중 타겟 조합
            → 모듈 G: 최종 보고서 생성

각 단계에서 파라미터가 어떻게 변환되고 전달되는지 추적해야 하지만:

  • 7개 모듈의 코드를 동시에 컨텍스트에 담을 수 없음
  • 3단계째 타고 들어가면 1단계에서 왜 시작했는지 맥락을 놓침
  • 각 모듈 경계에서 데이터가 파일/DB로 변환되면서 추적 끊김

이 문제는 코드가 부족해서가 아니라, 너무 많고 넓어서 발생한다.

해결: 시스템 메타데이터 (파이프라인 맵, 영향도 그래프, 모듈 간 데이터 계약)

1.4 2계층 메타데이터 아키텍처

두 유형의 문제를 해결하기 위해 2계층 구조를 설계했다.

계층 이름 범위 문서 수 해결하는 문제
모듈 메타데이터 Repository Metadata 레포 1개 레포당 1개 유형 1: 코드에 없는 도메인 지식
시스템 메타데이터 System Metadata 전체 시스템 1개 유형 2: 모듈 간 추적이 끊기는 정보

Agent는 질문의 범위에 따라 참조하는 계층이 달라진다:

질문 범위 참조 계층 예시
“이 변수가 왜 0.5인가?” 모듈 메타데이터 해당 모듈의 설계 의도 참조
“모듈 A 변경이 모듈 G에 미치는 영향?” 시스템 메타데이터 파이프라인 전체 영향도 그래프 참조

1.5 7개 범주 체계

v1의 13개 카테고리에서 유사 성격을 통합하여 7개 범주로 재편했다.

# 범주 정의 핵심 질문
1 I/O 규약 Input/Output 간 포맷, 구조, 값 범위 약속으로, 코드만으로 정확한 내용을 알 수 없는 것 “데이터가 오갈 때 지켜야 할 숨겨진 약속은?”
2 파이프라인 흐름과 상태 전이 실행 순서, 분기 조건, 상태 전이 규칙으로, 코드만으로 이유와 도메인 의미를 알 수 없는 것 “실행 순서와 분기의 숨겨진 이유는?”
3 설계 의도와 도메인 근거 상수, 임계값, 알고리즘 선택이 도메인 제약이나 실험 결과에서 비롯된 것으로, 코드만으로 알 수 없는 것 “왜 이 값, 이 규칙, 이 알고리즘인가?”
4 DB 스키마의 숨겨진 규약 암묵적 FK, NULL의 비즈니스 의미, 쿼리 성능 전제로, 코드만으로 알 수 없는 것 “DB 구조의 암묵적 관계와 의미는?”
5 실행 전제와 제약 라이브러리 버전, CLI 인자 관계, 메모리/디스크 한계로, 코드만으로 알 수 없는 것 “코드 실행에 필요한 숨겨진 조건은?”
6 운영 환경과 규약 test/prod 차이, 배포 절차, 로그 해석 규칙으로, 코드만으로 알 수 없는 것 “운영 시에만 알 수 있는 규칙은?”
7 코드 의미론적 함정 (CST) 변수명, 타입 힌트, 주석이 암시하는 의미와 실제 동작이 다른 곳 “코드 표면과 실제 동작이 다른 곳은?”

범주 1~6은 정보가 없어서 모르는 것이고, 범주 7은 정보가 있지만 틀려서 잘못 아는 것이다. 이 구분이 중요한 이유는, CST(Code Semantic Trap)는 Agent가 코드를 읽고 확신을 가지고 잘못된 답변을 할 수 있기 때문이다.

1.5.1 CST 예시

# 함수명이 'pass'를 포함하지만, True가 '통과'인지 '존재'인지 불명확
def _pass_dimer_check(seq1, seq2):
    ...
    return True  # 다이머 기준 '통과'? 다이머 '존재'?

# 호출부에서 혼용
if _pass_dimer_check(a, b):      # 통과했으니 계속?
if not _pass_dimer_check(a, b):  # 통과 못했으니 스킵?

Agent가 함수명만 보고 “pass = 통과”로 해석하면, 호출부의 의도를 정반대로 이해할 수 있다.

1.6 형식 변경: 테이블에서 문장으로

1.6.1 v1: 테이블 형식

| # | 파일 | 물리명 | 논리명 | 타입 | 값 | 의미 |
|---|------|--------|--------|------|-----|------|

1.6.2 v2: 문장 형식 + 확신도 태그

## 범주 3: 설계 의도와 도메인 근거

`ALLOW_OVERLAP_INIT = 0.5`: 오버랩 허용 초기 임계값.
짧은 올리고 길이의 50% 초과 겹침이면 중첩으로 판정.
[INFERRED: oligo.py:L91에서 overlap_length > shorter_length * criteria로 사용]
[UNKNOWN: 0.5가 생물학적 실험 결과에서 유래했는지, 관행적 기본값인지?]

1.6.3 왜 변경했는가

관점 테이블 형식 문장 형식
담당자 검토 셀 단위로 쪼개져 맥락 파악 어려움 문맥 있는 문장으로 이해 용이
환각 위험 빈 셀을 채우려고 LLM이 지어냄 모르면 [UNKNOWN]으로 표시
맥락 전달 “왜”와 “위험”이 분리됨 하나의 문단에 이유+위험+근거 통합
Agent 활용 후처리 없이 바로 사용 어려움 JSON 변환 후 구조화 검색 가능

1.6.4 확신도 태그 체계

태그 의미 담당자 액션
[FACT] 코드에서 직접 확인 가능. 맥락 제공용 최소한의 사실 검토 불필요
[INFERRED] 코드 패턴에서 추론. 맞을 가능성 높지만 확인 필요 확인 후 FACT 또는 수정
[UNKNOWN] 코드에서 절대 알 수 없음. 담당자만 아는 정보 반드시 작성 (핵심 가치)

UNKNOWN 태그가 메타데이터의 핵심 가치이다. LLM이 “모릅니다”라고 정직하게 표시한 항목이, 담당자가 채워야 할 진짜 메타데이터이기 때문이다.

}

1.8 테스트 추출 결과: v1 vs v2

동일 모듈에 대해 v1과 v2로 추출한 결과를 비교했다.

1.8.1 수량 비교

v1 (원자적) v2 (문장 형식)
항목 수 1,100개 258개 태그
형식 테이블 행 문장 + 확신도 태그
핵심 정보 비율 ~20% ~64% (INFERRED+UNKNOWN)

1.8.2 확신도 태그 분포

태그 개수 비율 의미
[FACT] 94 36% 코드에서 확인 가능 (맥락용)
[INFERRED] 88 34% 추론 (확인 필요)
[UNKNOWN] 76 30% 코드에서 알 수 없음 (담당자 필수)

담당자가 집중 검토해야 할 항목은 UNKNOWN 76개이다. v1에서는 1,100개를 전부 검토해야 했다.

1.8.3 추출 품질 예시

범주 1 (I/O 규약) - 침묵 실패 패턴:

SQLite에서 읽은 필드는 콤마로 구분된 다중 값일 수 있다. _process_sequence_data()에서 split(',')로 분리하여 각각을 독립 올리고로 처리한다. start, end, strand도 콤마 구분이므로 각 필드의 값 개수가 반드시 동일해야 한다. 코드에서는 zip()으로 묶으므로 개수 불일치 시 짧은 쪽에 맞춰 잘리며, 에러 없이 일부 데이터가 유실된다(침묵 실패). [INFERRED: preprocessing.py:L220-L228] [UNKNOWN: 상위 모듈이 콤마 구분 다중 값을 생성하는 정확한 조건은?]

이런 항목은 v1 테이블 형식으로는 표현할 수 없었다. “침묵 실패”라는 위험과 “왜 이런 구조인가”라는 맥락이 하나의 문단에 담겨 있다.

범주 7 (CST) - 함수명 의미 반전:

_pass_dimer_* 함수들: 함수명 ’pass’가 ’통과’를 암시하지만, True 반환이 ’다이머 기준 통과’인지 ’다이머 존재’인지 코드마다 다르다. 호출부에서 if _pass_dimer()와 if not _pass_dimer() 혼용. [INFERRED: dimer_calculator.py 호출 패턴에서 추론]

Agent가 이 메타데이터 없이 코드를 읽으면, 함수명에 속아 정반대로 해석할 수 있는 함정이다.

1.9 Agent 아키텍처 방향: 메타데이터를 어떻게 활용할 것인가

메타데이터를 구축한 후, Agent가 이를 활용하는 아키텍처는 단계적으로 진화시킬 수 있다. (상세는 RAG 아키텍처의 진화 참고)

[1단계] 기본 RAG
  - 메타데이터 JSON을 벡터 스토어에 인덱싱
  - 질문 → 벡터 검색 → 관련 메타데이터 → LLM 답변
  - 단순 질문에 적합 ("이 변수가 왜 0.5인가?")

[2단계] 에이전틱 RAG
  - Agent가 질문을 분석하여 다중 검색 수행
  - 메타데이터 JSON에 관계 필드(affects, affected_by) 추가
  - 다단계 추론 가능 ("모듈 A 변경이 모듈 G에 미치는 영향")

[3단계] 그래프 RAG
  - 메타데이터의 관계 정보를 그래프 DB로 마이그레이션
  - 깊은 영향도 분석, 파이프라인 전체 추적

어떤 단계에서든 메타데이터는 핵심 자산이다:

  • 기본 RAG: 메타데이터가 검색 대상
  • 에이전틱 RAG: Agent가 메타데이터를 참조하며 추론
  • 그래프 RAG: 메타데이터에서 추출한 관계가 그래프의 기반

1.10 LLM 기반 추출의 현실적 과제

1.10.1 품질 안정성

문제 설명
구조 불일치 같은 프롬프트인데 범주를 빠뜨리거나 다른 형식으로 출력
깊이 불일치 모듈마다 추출 깊이가 달라 비교 불가
FACT 과잉 이번 추출에서 FACT가 36%로 높았음 (프롬프트 억제 강화 필요)

1.10.2 유지보수

문제 설명
갱신 판단 어떤 코드 변경이 메타데이터 갱신을 필요로 하는가?
부분 갱신 전체를 다시 추출하지 않고, 변경된 부분만 갱신할 수 있는가?
핵심 정보 보존 담당자가 채워넣은 UNKNOWN 정보가 재추출 시 유실되면 안 됨

1.10.3 대응 방향

방향 설명
출력 검증 파이프라인 FACT 비율 체크, 범주 누락 확인, 형식 일관성 자동 검증
기준 모듈 벤치마크 프롬프트/모델 변경 시 동일 모듈로 재추출하여 품질 비교
담당자 피드백 루프 검토 결과를 프롬프트 개선에 체계적으로 반영

1.11 v1에서 v2로의 변화 요약

항목 v1 v2
메타데이터 정의 “사람만 아는 지식” (모호) “코드로 알 수 없고, 수정에 필수인 맥락” (명확)
판별 기준 프로그램이 자동 추출 가능한가? 이 정보 없이 수정하면 문제가 생기는가?
카테고리 13개 7개 범주
형식 테이블 행 (8열) 문장 + 확신도 태그
항목 수 1,100개 258개 태그
핵심 정보 비율 ~20% ~64%
담당자 부담 1,100개 전수 검토 UNKNOWN 76개 집중 검토
구조 단일 레포 메타데이터만 2계층 (모듈 + 시스템)

핵심 교훈: 많이 추출하는 것이 아니라, 정확히 필요한 것만 추출하는 것이 메타데이터의 가치를 높인다. 1,100개에서 258개로 줄었지만, Agent에게 실질적으로 도움이 되는 정보의 밀도는 3배 이상 높아졌다.
