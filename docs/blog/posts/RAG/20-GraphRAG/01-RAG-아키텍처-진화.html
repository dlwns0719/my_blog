1 RAG 아키텍처의 진화

RAG(Retrieval-Augmented Generation)는 LLM의 지식 한계를 외부 데이터 검색으로 보완하는 기술이다. 하지만 “검색 한 번으로 충분한가?”, “문서 간 관계를 어떻게 추적하는가?”라는 질문에 따라 RAG 아키텍처는 3단계로 진화해왔다.

기본 RAG → 에이전틱 RAG → 그래프 RAG
(단순 검색)  (다단계 추론)   (관계 기반 탐색)

1.1 기본 RAG (Naive RAG)

1.1.1 동작 원리

가장 단순한 형태의 RAG이다. 사용자 질문을 벡터로 변환하고, 유사한 문서 청크를 검색하여 LLM에 전달한다.

사용자 질문
  → 질문을 벡터로 변환 (Embedding)
    → 벡터 스토어에서 유사도 검색 (Retrieval)
      → 검색된 청크를 LLM에 전달
        → LLM이 답변 생성 (Generation)

1.1.2 저장 구조

문서를 일정 크기의 청크(chunk)로 분할하고, 각 청크를 벡터로 변환하여 저장한다.

원본 문서 → 청크 분할 → 임베딩 → 벡터 스토어

[청크1: "파라미터 X는 0.5로 설정되어 있다..."] → [0.23, 0.45, 0.12, ...]
[청크2: "모듈 A는 모듈 B의 출력을 입력으로..."] → [0.67, 0.33, 0.89, ...]
[청크3: "최적화 알고리즘은 greedy 방식으로..."] → [0.11, 0.78, 0.56, ...]

각 청크는 독립적이다. 청크1과 청크2 사이에 관계가 있더라도, 벡터 스토어는 그 관계를 모른다.

1.1.3 한계

한계 설명
단일 검색 검색이 1회로 끝남. 첫 검색에서 충분한 정보를 못 찾으면 답변 품질 저하
관계 무시 문서 간 관계를 모름. “A가 B에 영향을 준다”는 정보가 저장되지 않음
복잡한 질문 취약 여러 문서를 걸쳐 추론해야 하는 질문에 약함

적합한 경우: 단일 문서에서 답을 찾을 수 있는 단순 질의응답 (FAQ, 문서 검색 등)

1.2 에이전틱 RAG (Agentic RAG)

1.2.1 기본 RAG의 어떤 한계를 해결하는가

기본 RAG는 검색 1회로 끝난다. 하지만 현실의 질문은 한 번의 검색으로 답하기 어려운 경우가 많다.

“모듈 A에서 설계 조건을 변경하면, 최종 출력 모듈 G에 어떤 영향이 있는가?”

이 질문에 답하려면 A → B → C → … → G까지 단계적으로 정보를 수집해야 한다. 기본 RAG는 이런 다단계 추론(multi-hop reasoning)을 할 수 없다.

1.2.2 동작 원리

LLM이 스스로 판단하여 여러 번 검색한다. 검색 결과를 보고, 추가 검색이 필요한지, 어디를 더 봐야 하는지를 LLM이 결정한다.

사용자 질문: "모듈 A의 파라미터 변경이 모듈 G 출력에 미치는 영향은?"

Agent(LLM):
  "이 질문은 파이프라인 전체를 추적해야 한다"
  → 1차 검색: 모듈 A 관련 문서 검색
    → "A의 출력이 모듈 B의 입력이 된다"
  → 2차 검색: 모듈 B 관련 문서 검색
    → "B는 C에 필터링된 결과를 전달한다"
  → 3차 검색: 모듈 C 관련 문서 검색
    → ...
  → N차: 모듈 G까지 도달
  → 종합하여 영향 경로 답변 생성

1.2.3 핵심 구성 요소

구성 요소 역할
메인 Agent 질문을 분석하고, 검색 전략을 수립하고, 결과를 종합하여 답변
서브 Agent (선택) 특정 도메인/모듈을 전담 검색하는 하위 Agent
도구 (Tools) 벡터 검색, 키워드 검색, DB 조회 등 Agent가 사용할 수 있는 검색 도구
메모리 이전 검색 결과를 기억하여 중복 검색 방지

1.2.4 서브 Agent 패턴

복잡한 시스템에서는 메인 Agent가 서브 Agent에게 작업을 분배한다.

메인 Agent: "파이프라인 전체를 추적해야 하니, 각 모듈 담당에게 물어보자"
  → 서브 Agent 1 (모듈 A 전담): A의 출력 구조 검색
  → 서브 Agent 2 (모듈 B 전담): B의 입력 처리 방식 검색
  → 서브 Agent 3 (모듈 C 전담): C의 변환 로직 검색
  → ...
메인 Agent: 서브 Agent들의 결과를 종합하여 최종 답변

1.2.5 기본 RAG와의 차이

기본 RAG 에이전틱 RAG
검색 횟수 1회 고정 Agent가 필요한 만큼 반복
판단 주체 개발자가 미리 설계한 파이프라인 LLM이 실시간으로 판단
검색 전략 고정 (항상 같은 방식) 질문에 따라 전략 변경
복잡한 질문 한 번에 못 찾으면 부정확한 답변 단계적 추적 가능
비용 LLM 호출 1회 LLM 호출 N회 (비용 증가)

1.2.6 한계

에이전틱 RAG도 결국 텍스트 검색에 의존한다. “A → B → C” 관계를 추적할 때, 각 단계에서 텍스트를 검색하고 LLM이 관계를 추론해야 한다. 관계가 명시적으로 저장되어 있지 않기 때문에:

  • 검색 결과에 관계 정보가 포함되지 않을 수 있음
  • 경로가 길어질수록 중간에 맥락을 놓칠 수 있음
  • 매번 LLM이 관계를 추론하므로 비용과 시간이 증가

적합한 경우: 다단계 추론이 필요하지만, 관계 경로가 5단계 이내인 질문. 대부분의 실무 시나리오에서 충분히 효과적이다.

1.3 그래프 RAG (Graph RAG)

1.3.1 에이전틱 RAG의 어떤 한계를 해결하는가

에이전틱 RAG는 관계를 텍스트에서 매번 추론한다. 하지만 데이터를 저장할 때 관계 자체를 함께 저장하면 어떨까? 그래프 RAG의 핵심 아이디어이다.

1.3.2 핵심 개념: 노드와 엣지

그래프 RAG는 데이터를 노드(Node)와 엣지(Edge)로 저장한다.

노드: 개별 개념, 엔티티, 코드 요소 등
엣지: 노드 간의 관계 (방향 + 의미)

예시:
  (파라미터 X) --[사용됨]--> (필터링 함수)
  (필터링 함수) --[출력]--> (중간 결과 테이블)
  (중간 결과 테이블) --[입력됨]--> (최적화 모듈)
  (최적화 모듈) --[생성]--> (최종 보고서)

1.3.3 저장 구조 비교

일반 RAG의 저장:
  [청크1] → 벡터  ← 독립적, 관계 없음
  [청크2] → 벡터
  [청크3] → 벡터

그래프 RAG의 저장:
  [노드: 파라미터 X] → 벡터 + 관계 정보
       ↓ (사용됨)
  [노드: 필터링 함수] → 벡터 + 관계 정보
       ↓ (출력)
  [노드: 중간 결과] → 벡터 + 관계 정보
       ↓ (입력됨)
  [노드: 최적화 모듈] → 벡터 + 관계 정보

벡터 스토어는 하나이고, 그 안에 데이터를 넣을 때 엣지(관계) 정보를 함께 저장하는 것이 핵심 차이이다.

1.3.4 검색 방식의 차이

질문: "파라미터 X를 변경하면 최종 출력에 어떤 영향?"

일반 RAG:
  → "파라미터 X"가 언급된 청크 3개 반환
  → LLM이 그 3개만 보고 답변 (영향 경로를 모름)

에이전틱 RAG:
  → 1차: "파라미터 X" 검색 → "필터링 함수에서 사용"
  → 2차: "필터링 함수" 검색 → "중간 결과를 출력"
  → 3차: "중간 결과" 검색 → "최적화 모듈에 입력"
  → 4차: "최적화 모듈" 검색 → "최종 보고서 생성"
  → LLM이 4번의 검색 결과를 종합 (비용 4배)

그래프 RAG:
  → "파라미터 X" 노드 찾음
  → 연결된 엣지를 따라 그래프 탐색 (DB 레벨에서 수행)
    → 필터링 함수 → 중간 결과 → 최적화 모듈 → 최종 보고서
  → 영향 경로 전체를 한 번에 LLM에 전달
  → LLM 호출 1회로 완전한 답변 가능

1.3.5 구현 아키텍처

그래프 RAG는 그래프 DB + 벡터 스토어를 결합하여 구현한다.

[그래프 DB]                         [벡터 스토어]
 노드와 관계를 저장                    텍스트/벡터를 저장
 관계 기반 탐색 수행                   의미 기반 유사도 검색

 (A)--[영향]-->(B)-->(C)            "파라미터 X는..." → [0.23, ...]

              ↘          ↙
             [Agent / LLM]
      그래프 탐색 결과 + 벡터 검색 결과를 종합

1.3.5.1 주요 기술 스택

역할 도구/서비스
그래프 DB Neo4j, Amazon Neptune, Azure Cosmos DB (Gremlin API)
벡터 스토어 FAISS, Chroma, Azure AI Search, Pinecone
LLM GPT-4, Claude, Azure OpenAI
프레임워크 LangChain, LlamaIndex, Microsoft GraphRAG

1.3.5.2 검색 흐름 상세

# 개념적인 그래프 RAG 검색 흐름 (pseudo-code)

# 1단계: 질문에서 핵심 엔티티 추출 (LLM 사용)
entities = llm.extract_entities("파라미터 X를 변경하면 최종 출력에 어떤 영향?")
# → ["파라미터 X"]

# 2단계: 벡터 검색으로 관련 노드 찾기
start_nodes = vector_store.similarity_search("파라미터 X")
# → [Node("파라미터_X", id=42)]

# 3단계: 그래프 DB에서 관계 탐색
impact_path = graph_db.traverse(
    start_node=42,
    relationship_types=["사용됨", "출력", "입력됨"],
    max_depth=5
)
# → [파라미터_X → 필터링_함수 → 중간_결과 → 최적화_모듈 → 최종_보고서]

# 4단계: 경로의 각 노드에서 상세 컨텍스트 수집
context = []
for node in impact_path:
    details = vector_store.get_document(node.id)
    context.append(details)

# 5단계: LLM에 경로 + 컨텍스트 전달하여 답변 생성
answer = llm.generate(
    question="파라미터 X를 변경하면 최종 출력에 어떤 영향?",
    graph_path=impact_path,
    context=context
)

1.3.6 그래프 구축 방법

그래프 RAG에서 가장 어려운 부분은 그래프를 어떻게 만드는가이다. 두 가지 접근법이 있다.

1.3.6.1 방법 1: LLM 기반 자동 추출

문서나 코드에서 LLM이 엔티티와 관계를 자동으로 추출한다.

원본 텍스트: "함수 calculate_score()는 threshold 값을 사용하여
             후보를 필터링하고, 결과를 result_table에 저장한다."

LLM 추출 결과:
  노드: [calculate_score, threshold, result_table]
  엣지: [(threshold) --[파라미터]--> (calculate_score)]
        [(calculate_score) --[저장]--> (result_table)]

Microsoft GraphRAG 라이브러리가 이 방식을 사용한다.

1.3.6.2 방법 2: 수동/반수동 정의

도메인 전문가가 핵심 관계를 직접 정의한다. 정확도가 높지만 노력이 많이 든다.

# 수동으로 정의한 파이프라인 관계
nodes:
  - id: module_a
    type: module
    description: "입력 데이터 전처리"
  - id: module_b
    type: module
    description: "필터링 및 정렬"

edges:
  - source: module_a
    target: module_b
    relationship: "출력을_입력으로_전달"
    data_format: "SQLite DB"
    key_tables: ["sequence_table", "config_table"]

1.3.6.3 방법 3: 하이브리드

LLM이 1차 추출하고, 전문가가 검증/보완한다. 실무에서 가장 현실적인 접근이다.

1.4 3가지 아키텍처 종합 비교

기본 RAG 에이전틱 RAG 그래프 RAG
검색 방식 벡터 유사도 1회 Agent가 다중 검색 그래프 탐색 + 벡터 검색
관계 추적 불가 LLM이 매번 추론 그래프 DB에서 직접 탐색
복잡한 질문 약함 중간 (5단계 이내) 강함 (깊은 경로도 가능)
구축 난이도 낮음 중간 높음
운영 비용 낮음 중간 (LLM 다중 호출) 높음 (그래프 DB 운영)
LLM 호출 수 1회 N회 (검색 횟수만큼) 1~2회
저장소 벡터 스토어 벡터 스토어 벡터 스토어 + 그래프 DB
적합한 시나리오 단순 Q&A 다단계 추론 영향도 분석, 관계 탐색

1.5 그래프 DB 없이 유사하게 구현하는 방법

그래프 DB를 도입하기 전에, 벡터 스토어만으로 관계 기반 검색을 흉내내는 방법이 있다.

1.5.1 메타데이터 필드에 관계 정보 포함

{
  "id": "param-001",
  "title": "파라미터 X",
  "content": "필터링 임계값으로 사용되는 파라미터...",
  "affects": ["func-002", "func-003"],
  "affected_by": ["config-001"],
  "pipeline_stage": "stage-2",
  "module": "module-a"
}

Agent가 검색 결과의 affects 필드를 읽고, 해당 ID를 추가 검색하는 방식이다.

1차 검색: "파라미터 X" → param-001 반환
  → affects: ["func-002", "func-003"]
2차 검색: func-002 조회 → affects: ["result-004"]
3차 검색: result-004 조회 → affects: ["output-005"]
→ 경로 완성

이것은 사실상 에이전틱 RAG + 관계 메타데이터의 조합이다. 그래프 DB 없이도 중간 규모의 관계 탐색이 가능하다.

1.5.2 그래프 DB와의 비교

그래프 DB 방식 메타데이터 필드 방식
관계 탐색 DB 엔진이 수행 (빠름) Agent가 다중 검색 (느림)
구축 난이도 높음 (그래프 DB 설계/운영) 낮음 (JSON 필드 추가)
복잡한 경로 10단계 이상도 가능 3~4단계까지 현실적
추가 비용 그래프 DB 비용 발생 벡터 스토어 비용만
전환 용이성 - 관계 데이터가 축적되어 그래프 DB 전환 시 재활용 가능

1.6 단계별 도입 전략

실무에서는 처음부터 그래프 RAG를 구축하기보다, 단계적으로 진화시키는 것이 현실적이다.

[1단계] 기본 RAG
  - 벡터 스토어에 문서/메타데이터 저장
  - 단순 질의응답 서비스 구축
  - 검색 패턴과 사용자 질문 유형 분석
      |
      | 어떤 질문이 자주 실패하는지 파악
      v
[2단계] 에이전틱 RAG
  - Agent가 다중 검색을 수행하도록 확장
  - 메타데이터 JSON에 관계 필드(affects, affected_by) 추가
  - 그래프 DB 없이도 관계 기반 검색 가능
      |
      | 관계 데이터 축적, 탐색 깊이/빈도 증가
      v
[3단계] 그래프 RAG
  - 축적된 관계 데이터를 그래프 DB로 마이그레이션
  - 깊은 경로 탐색, 영향도 분석 본격 지원
  - 그래프 시각화로 시스템 이해도 향상

1.6.1 전환 판단 기준

기준 2단계 유지 3단계 전환
관계 탐색 깊이 3~4단계면 충분 5단계 이상 빈번
관계 데이터 규모 수백 개 수천 개 이상
탐색 빈도 가끔 핵심 기능으로 상시 사용
성능 요구 응답 수 초 허용 실시간 응답 필요
팀 역량 그래프 DB 운영 경험 없음 운영 가능

1.7 Azure 서비스 매핑 (참고)

각 아키텍처 단계에서 활용할 수 있는 Azure 서비스를 정리한다.

1.7.1 1단계: 기본 RAG

역할 Azure 서비스 설명
벡터 스토어 + 검색 Azure AI Search 벡터 검색 + 키워드 검색 + 시맨틱 랭킹을 하나의 서비스로 제공
LLM Azure OpenAI GPT-4o, GPT-4 등 OpenAI 모델을 Azure 인프라에서 사용
오케스트레이션 Azure AI Foundry RAG 파이프라인 구성, 프롬프트 관리, 평가 도구

1.7.2 2단계: 에이전틱 RAG

역할 Azure 서비스 설명
Agent 실행 Azure Container Apps Agent 애플리케이션을 서버리스 컨테이너로 실행. 사용량에 따라 자동 확장/축소
API 관리 Azure API Management Agent API의 인증, 사용량 제한, 로깅, 버전 관리
LLM 라우팅 Azure AI Gateway 여러 LLM 모델을 하나의 엔드포인트로 관리. 모델 선택, 비용 추적, 폴백(fallback) 처리

1.7.3 3단계: 그래프 RAG

역할 Azure 서비스 설명
그래프 DB Azure Cosmos DB (Gremlin API) Apache TinkerPop Gremlin 쿼리 언어를 지원하는 분산 그래프 데이터베이스
그래프 DB (대안) Azure Cosmos DB (Apache Gremlin) 글로벌 분산, 자동 확장, 밀리초 응답 시간
벡터 + 그래프 통합 검색 AI Search + Cosmos DB 조합 벡터 검색 결과를 시작점으로, 그래프 DB에서 관계 탐색

1.7.4 Azure 서비스 용어 해설

서비스 한 줄 설명
Azure AI Search 문서를 인덱싱하고 벡터/키워드/시맨틱 검색을 제공하는 관리형 검색 엔진
Azure OpenAI OpenAI 모델(GPT-4 등)을 Azure 클라우드에서 API로 사용하는 서비스
Azure Container Apps Docker 컨테이너를 서버리스로 실행하는 서비스. 서버 관리 없이 앱 배포 가능
Azure API Management API 앞단에서 인증, 속도 제한, 모니터링을 처리하는 게이트웨이
Azure AI Gateway 여러 AI 모델 호출을 하나의 창구로 통합 관리. 비용 추적, 모델 폴백, 로깅
Azure Cosmos DB 글로벌 분산 NoSQL/그래프 데이터베이스. Gremlin API로 그래프 쿼리 지원
Azure AI Foundry AI 애플리케이션 개발/배포/모니터링 통합 플랫폼 (구 Azure AI Studio)

1.8 정리

  • 기본 RAG: 벡터 검색 1회. 단순 Q&A에 적합. 빠르게 구축 가능.
  • 에이전틱 RAG: Agent가 다중 검색. 다단계 추론 가능. 대부분의 실무 시나리오에 충분.
  • 그래프 RAG: 관계를 명시적으로 저장. 깊은 영향도 분석에 강함. 구축/운영 비용 높음.

핵심 메시지: 1단계부터 시작하고, 관계 데이터를 축적하면서 필요에 따라 진화시키는 것이 가장 현실적인 전략이다. 처음부터 그래프 RAG를 목표로 하되, 그래프 DB 도입은 실제 필요성이 검증된 후에 해도 늦지 않다.
